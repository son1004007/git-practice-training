# 00. Git 교육 개요

## 1. 교육 목적

이 교육은 Git을 처음 사용하는 구성원이 실무에서 필요한 최소 흐름을 익히는 것을 목표로 합니다.

단순히 명령어를 외우는 것이 아니라, 실제 업무 산출물을 Git으로 관리하는 흐름을 반복합니다.

```text
수정 → 변경 확인 → stage → commit → push → GitHub 확인
```

---

## 2. 교육 대상

- 데이터 분석가
- SQL, Python, 보고서 산출물을 관리해야 하는 실무자
- GitHub 저장소를 처음 사용하는 구성원
- VS Code 기반으로 Git 작업을 수행해야 하는 구성원

---

## 3. 교육 완료 기준

교육 후 아래 작업을 스스로 수행할 수 있어야 합니다.

| 구분 | 완료 기준 |
|---|---|
| 환경 구성 | Git Bash와 VS Code 설치 완료 |
| Git 설정 | user.name, user.email 설정 완료 |
| 저장소 사용 | GitHub 레포 clone 가능 |
| 변경 관리 | add, commit, push 가능 |
| 작성규칙 | `type(scope): 변경 내용` 형식으로 commit 작성 가능 |
| 협업 흐름 | branch 생성 후 Pull Request 생성 가능 |
| 실무 적용 | SQL, Python, 보고서 변경 이력 관리 가능 |

---

## 4. 교육 진행 순서

| 순서 | 주제 | 실습 결과 |
|---:|---|---|
| 1 | Git Bash 설치 | Git 명령어 실행 가능 |
| 2 | VS Code 설치 | Git 변경사항을 GUI로 확인 가능 |
| 3 | GitHub 개인 레포 생성 | 개인 실습 저장소 준비 |
| 4 | clone 실습 | 원격 저장소를 로컬로 복제 |
| 5 | commit/push 실습 | 변경사항을 GitHub에 반영 |
| 6 | commit message 규칙 | 사내 작성규칙 적용 |
| 7 | branch/PR 실습 | 협업 흐름 이해 |
| 8 | 실무 시나리오 | SQL, Python, 보고서 변경 관리 |

---

## 5. 실무에서 Git을 사용하는 이유

Git은 단순 백업 도구가 아닙니다.

다음 질문에 답하기 위한 변경 이력 관리 도구입니다.

```text
누가 변경했는가?
언제 변경했는가?
무엇을 변경했는가?
왜 변경했는가?
이 변경이 분석 결과에 영향을 주는가?
문제가 생기면 어느 시점으로 되돌릴 수 있는가?
```

---

## 6. 교육 운영 원칙

- 처음부터 모든 명령어를 외우지 않습니다.
- 실무에서 자주 쓰는 흐름만 먼저 익힙니다.
- 하나의 commit에는 하나의 목적만 담습니다.
- commit message는 사내 작성규칙을 사용합니다.
- 실습 파일은 작게 수정하고 자주 commit합니다.

---

## 7. 최소 암기 명령어

```bash
git clone <repository-url>
git status
git add <file>
git commit -m "type(scope): 변경 내용"
git push origin <branch>
git pull origin main
git switch -c <branch-name>
```

---

## 8. 핵심 원칙

```text
Git 실습의 핵심은 명령어 암기가 아니라 변경 이력의 품질입니다.
```

좋은 Git 이력은 나중에 분석 기준, 코드 변경, 보고서 수정 이유를 빠르게 확인할 수 있게 해줍니다.
