# 08. 자주 발생하는 문제

## 1. 저장소 clone이 되지 않는 경우

### 원인

- 저장소 주소가 잘못됨
- GitHub 로그인 또는 인증 문제
- 네트워크 차단
- 저장소 접근 권한 없음

### 해결

```bash
git clone 저장소주소
```

확인할 것:

- GitHub 저장소 주소가 올바른지 확인
- 저장소가 Private이면 접근 권한 확인
- 회사망에서 GitHub 접근이 가능한지 확인

---

## 2. commit이 되지 않는 경우

### 원인

- stage된 파일이 없음
- Git 사용자 설정이 없음

### 해결

```bash
git status
git add 파일명
git commit -m "docs(memo): 업무 메모 수정"
```

사용자 설정이 없으면 Git 기본 설정을 먼저 수행합니다.

---

## 3. push가 되지 않는 경우

### 원인

- 원격 저장소 인증 문제
- 브랜치 이름 오류
- 원격 저장소에 먼저 반영된 변경사항 존재

### 해결

현재 브랜치를 확인합니다.

```bash
git branch
```

원격 저장소 최신 내용을 먼저 가져옵니다.

```bash
git pull origin main
```

다시 push합니다.

```bash
git push origin 현재브랜치명
```

---

## 4. GitHub에 변경사항이 보이지 않는 경우

### 원인

- commit만 하고 push하지 않음
- 다른 브랜치에 push함
- GitHub 화면에서 branch 선택이 다름

### 해결

```bash
git status
git branch
git log --oneline -5
```

확인할 것:

- push 여부
- branch 이름
- GitHub 화면의 branch 선택값

---

## 5. conflict가 발생한 경우

### 원인

같은 파일의 같은 부분을 여러 사람이 수정했습니다.

### 해결 흐름

```text
충돌 파일 열기
→ 충돌 표시 확인
→ 필요한 내용만 남기기
→ 파일 저장
→ add
→ commit
```

명령어:

```bash
git status
git add 충돌파일명
git commit -m "fix(conflict): 충돌 내용 정리"
```

---

## 6. commit message를 잘못 작성한 경우

### 마지막 commit만 수정하는 경우

```bash
git commit --amend -m "docs(readme): 교육 개요 수정"
```

주의:

이미 여러 사람이 공유한 commit은 임의로 수정하지 않는 것이 안전합니다.

---

## 7. 잘못 수정한 파일을 되돌리고 싶은 경우

아직 commit하지 않은 파일을 되돌립니다.

```bash
git restore 파일명
```

stage한 파일을 stage에서 제거합니다.

```bash
git restore --staged 파일명
```

---

## 8. 실무 기준 대응 순서

문제가 발생하면 아래 순서로 확인합니다.

```text
1. git status 확인
2. 현재 branch 확인
3. git log 확인
4. GitHub branch 확인
5. pull 필요 여부 확인
6. 다시 add, commit, push 수행
```

---

## 9. 최소 확인 명령어

```bash
git status
git branch
git log --oneline -5
git remote -v
```

위 4개 명령어만 확인해도 대부분의 Git 문제 원인을 파악할 수 있습니다.
